# PLATFORM_AUDIT.md

Everything a green-field team needs to reproduce Elevra's platform infrastructure. Each section: what exists, where the code is, and what the tradeoffs were.

## 1. Auth

**Provider:** Supabase Auth.

**Client:** `src/lib/supabase/client.ts` — a single `createClient(SUPABASE_URL, SUPABASE_ANON_KEY)` instance imported by every Supabase-backed service. The URL and anon key ship in the client bundle (both are public — Row-Level Security in Postgres is what gates data access).

**Sign-in methods:**
- Email + password.
- Magic link.
- OAuth: Google, Apple, Microsoft, Outlook (`SocialAuthButton` UI + Supabase provider config).

**Session shape (`src/app/state/session.ts`):**
```ts
interface SessionIdentity {
  role: 'coach' | 'client';
  userId: string;                 // auth.users.id
  clientId?: string;              // clients.id — set when role === 'client'
  coachId?: string;               // coaches.id — set when role === 'coach'
  displayName: string;
  email: string;
  avatarUrl?: string;
  pronouns?: string;
  bio?: string;
  timezone?: string;
  useInitials?: boolean;
  emailVerifiedAt?: string;
}
```

The `userId` (auth) and `clientId` / `coachId` (domain) are intentionally separate — a single Supabase user could theoretically have both a client and coach row, though the current app doesn't surface that flow.

Persistence: Zustand `persist` under `elevra:session`. Session restore on cold boot happens through Supabase's own token-refresh in tandem with our local snapshot for fast render.

**Sign-out:** `services.session.signOut()` clears the store + calls `supabase.auth.signOut()` + navigates to `/auth/sign-in`.

**Password recovery:** `services.auth.sendPasswordResetEmail(email)` → Supabase-managed email → link routes to `/auth/reset-password?token=…`.

**Deep links:**
- `/auth/reset-password` — password reset landing.
- `/auth/signup-client/:code` — client signup via invite code (validated server-side via `api/send-invite.mjs` at issue time).
- `/invite/:code` — coach invite link.
- `/c/:slug` — public coach profile (unauthenticated).

**Two-factor:** `ManageSessionsSheet` + `TwoFactorSheet` in `src/app/shells/`. Backed by Supabase's TOTP MFA. Optional per-user.

**Account deletion:** `AccountDeleteModal` → confirms → posts to `api/account-delete.mjs` → server function anonymises client rows and hard-deletes user rows from `auth.users`. Cascade / anonymisation logic detailed in migration `0086_account_delete_cascade.sql`.

**Account export:** GDPR data export via `api/account-export.mjs` → server function assembles a signed zip of user's JSON records + media links, mailed to the user.

## 2. Roles & scope

Two roles: `coach` and `client`. The `Role` enum is not in the DB — it's implicit in which of `coaches.id` or `clients.id` is linked to the auth user (`user_id` FK columns on both tables).

`useCurrentCoachId()` / `useCurrentClientId()` from `src/app/state/session.ts` are the canonical accessors. Screens that require one role check via these; the router prevents the wrong role reaching a route via a Suspense-boundary route-guard component in the shell.

**Row-Level Security:** every domain table has a policy that gates rows by `auth.uid()` matching either `client.user_id` for that row, or the client's assigned coach's `user_id`. See `supabase/migrations/0002_rls_policies.sql` for the full policy set. New tables must add a corresponding policy.

## 3. Data layer

### Supabase client

`src/lib/supabase/client.ts`. Single instance. Generic `Database` type is auto-generated by `supabase gen types typescript` — file at `src/lib/supabase/types.gen.ts`.

### Service pattern

Every domain has a Supabase service file at `src/services/supabase/<domain>.ts`. The shape is:

```ts
export function buildSupabase<Domain>Service(): <Domain>Service {
  return {
    list: async (params) => { ... },
    get: async (id) => { ... },
    create: async (input) => { ... },
    update: async (id, patch) => { ... },
    // ...
  };
}
```

Registered in `src/services/index.tsx` → wrapped in a React context via `<ServicesProvider>` in AppBootstrap → consumed via `useServices()`.

### Domain services (30+ services)

From `src/services/index.tsx`:
- `auth`, `clients`, `programmes`, `library`, `workouts`, `progress`, `food`, `hydration`, `checkins`, `wellbeing`, `wearables`, `cycle`, `messages`, `alerts`, `booking`, `business`, `notes`, `insights`, `telemetry`, `locations`, `gymScans`, `substitution`, `nudges`, `coachAlerts`, `availability`, `presence`, `clientTags`, `ratings`, `levels`.
- Built-in (non-Supabase): `pwa`, `aiCoach`, `taxonomy`.

Interfaces exported from `src/services/interfaces.ts`. Zod schemas for validation at boundaries in `src/services/schemas.ts`.

### Real-time

Supabase Realtime channels used for:
- **Messages** — `messages` table changes push to subscribed conversation participants.
- **Presence** — `presence` service tracks who's online per coach ↔ client pairing.
- **Alerts** — coach `coach_alerts` insertions push to the coach's alert bell in real time.

Implementation: `supabase.channel('name').on('postgres_changes', {…}, callback).subscribe()`.

## 4. State

**Global state:** Zustand. Stores under `src/app/state/`:
- `session.ts` — the signed-in identity.
- `theme.ts` — theme preference (`system` / `dark` / `light`).
- `a11y.ts` — motion + high-contrast preferences.
- `density.ts`, `font.ts`, `accent.ts` — user preferences.
- `messages.ts` — messages metadata (unread counts).
- `notificationPrefs.ts` — push preferences.
- `pinnedActions.ts` — dashboard prefs.
- `snoozePrefs.ts`, `snoozeHistory.ts` — snoozed alerts.
- ~40 total. Each is a tightly-scoped store persisted to localStorage under a distinct key.

**Ephemeral state:** `useState`, `useReducer`, and a handful of custom hooks (`useAsyncData`, `useAutoResize`, `useHotkey`, `useIsWideViewport`, `useLongPress`, `useAutoPopupSlot`).

**Server-state cache:** none currently. `useAsyncData` at `src/lib/useAsyncData.ts` is a thin fetch-once-per-mount wrapper. There's no react-query / SWR. A future rebuild should evaluate whether that's a decision to copy — see WEAKNESSES.md.

## 5. Offline

### Detection

`navigator.onLine` + a listener in `App.tsx` that surfaces the `<OfflinePill>` when offline. Copy: "Offline, showing your latest."

### Queue

`src/lib/offlineQueue.ts` — in-memory FIFO. `queueOrRun(key, thunk)`:
- If online: runs immediately.
- If offline: pushes to the queue, emits event, returns `Promise<undefined>`.

On reconnect, `flushQueue()` runs each thunk in order. Idempotency keys prevent double-writes when both the queued thunk and the user's subsequent live request fire.

**Persistence:** the queue is in-memory (documented). An IndexedDB-backed persistence layer is on the roadmap — see WEAKNESSES.md. Losing the queue on hard refresh is a documented risk.

### Photo store

`src/lib/photoStore.ts` — `idb-keyval`-backed IndexedDB store for photo blobs (client progress photos, food photos, gym scans). Photos are captured → stored to IDB → uploaded to Supabase Storage on the next online tick. Preview URLs come from `URL.createObjectURL(blob)` for immediate display.

`src/lib/gymPhotoStore.ts` — same pattern for gym scan photos.

### Service worker cache

`src/sw.ts` (custom, injected via `vite-plugin-pwa` with `strategies: 'injectManifest'`):
- **Precache:** `**/*.{js,css,html,svg,png,woff2}` from the build. `maximumFileSizeToCacheInBytes: 4 MiB` (raised from workbox's 2 MiB default because the bundled 873-exercise corpus pushes the main chunk over 2 MiB — long-term fix is to code-split `built-ins.ts`).
- **Fonts:** `CacheFirst`, `elevra-fonts` cache, 365-day expiry, 30 entries max.
- **Media:** `CacheFirst`, `elevra-media` cache.
- **GitHub CDN exercise frames:** `CacheFirst` targeted at `githubusercontent.com` so `LoopingFrames` demos work offline after first view.

### Push notification handling

SW `push` event → `showNotification(title, { body, data: { deepLink }, icon })`. SW `notificationclick` → focus an existing tab if one matches the scope, navigate to `deepLink`; else open a fresh tab.

## 6. Sync

**Optimistic UI:** most mutations write locally then `await` the Supabase call. On failure, a Toast surfaces and the local update rolls back.

**Long-poll / heartbeat:** not used. Real-time uses Supabase Realtime channels.

**Merge conflicts:** last-write-wins per row. Not sophisticated — a rebuild should consider CRDT-style merge for shared state (rare in this app).

**Background sync:** Web Sync API is not wired. The queue drains on `online` event, which requires the app to be open.

## 7. Push notifications

**Stack:** VAPID web push via `web-push` npm library.

**Client:**
- `src/lib/notifications.ts` orchestrates subscribe / unsubscribe.
- `PrePermissionSheet` walks the user through consent + category prefs.
- Subscription is written to a `push_subscriptions` table (endpoint, keys, user_id).

**Server:**
- `api/push-notify.mjs` — Vercel serverless function. Takes a payload + a set of user_ids, looks up their `push_subscriptions`, calls `web-push.sendNotification` for each.
- Triggers: cron jobs (see § 12), alert transitions, direct coach → client nudges.

**Categories** (per-user opt-in / opt-out):
- Programme reminders
- Check-in prompts
- Message replies
- Coach nudges
- System / update

**Cadence rules:**
- Never more than one message-reply notification per 30-minute window per pair (server-side rate limit).
- Cron-based reminders scheduled per user's `timezone`.
- Silent hours per user prefs.

## 8. Media

### Photos

- Captured via `PhotoCapture` (file input `capture="environment"`).
- Stored as Blob → IDB (`photoStore`) → uploaded to Supabase Storage bucket `photos` on next online tick.
- Public bucket for coach avatars + client avatars; private bucket for progress photos + food photos + gym scans (signed-URL access via `supabase.storage.from(bucket).createSignedUrl(path, 3600)`).

### Video

- Captured via `VideoRecorder` (file input `capture="environment"`, 60s cap).
- Direct upload to `exercise-media` bucket. No client-side compression (Phase 5 polish).

### Audio

- Captured via `VoiceRecorder` using `MediaRecorder` where available.
- Direct upload to `voice-notes` bucket.

### Signed URL lifetime

Default 1 hour. Regenerated on view. No aggressive caching so a shared image link expires quickly.

## 9. Search

### Global (coach only)

`SearchSheet` in `src/app/shells/`. Opens on `/` hotkey. Full-text search across:
- Clients (name, email, tags)
- Messages (conversation participants + last message snippet)
- Exercises (name, tags, muscles)
- Templates (name, description)

Search implementation: server-side `ilike` queries against Postgres. No dedicated search index — pg_trgm extension enabled on the largest tables.

### In-context search

Every list has a `SearchBar` for local filtering. For lists under ~500 items, filter is client-side. For lists that grow (library exercises), the SearchBar dispatches a server-side query.

## 10. Payments

**Not implemented in the current app.** A `subscription` table stub exists in the schema for future Stripe / paddle integration. `services.business` includes revenue metrics reads for hypothetical subscription data.

## 11. Analytics

**No external analytics SDK.**

**In-house telemetry:** `services.telemetry.record({ userId, kind, data })` writes to a `telemetry_events` table via Supabase. Kind names are `snake_case` domain events. Queryable via any admin with SQL access.

Categories:
- `app.*` — session, install, first-launch
- `nav.*` — route visits
- `feature.*` — feature usage
- `error.*` — client-side error taxonomy
- `experiment.*` — A/B (currently unused)

## 12. Errors

### Client-side

- `console.error` for developer signal.
- `RouteErrorBoundary` catches render errors at the route level → EmptyState with reload CTA.
- Reload-guard key (`RELOAD_GUARD_KEY`) in sessionStorage prevents infinite reload loops if the error fires on remount.
- `src/lib/supabase/error.ts` — maps Postgres error codes to VOICE-safe user copy.

### Server-side

- Vercel serverless function errors log to Vercel's console.
- No Sentry / Rollbar / Datadog. Vercel logs are the only telemetry.

## 13. Feature flags

**No structured system.** Two ad-hoc gates:

1. **`/dev/*` routes** — DEV mode OR `?dev=1` OR `localStorage['elevra:dev-unlock'] === '1'`.
2. **Per-feature Zustand booleans** — a few (e.g. `useCoachHomeLayout` for the new dashboard) live as persisted preferences.

Rebuild recommendation: use LaunchDarkly or Statsig if flags become a habit.

## 14. Deep linking

Route table in `src/app/App.tsx`. Every route is lazy-loaded. Deep links (from notifications, share sheets) navigate via `useNavigate()(deepLink)`.

**Universal links:** the PWA manifest scope is `/`, `start_url: '/?source=pwa'`. Links to `thesalesprogressor.co.uk/…` (marketing) don't reach the app; there's no `apple-app-site-association` file because there's no native shell.

## 16. Cron / background jobs

Vercel scheduled functions (defined in `vercel.json`):

- `api/cron-tick.mjs` — every 5 minutes. Fires programme reminders due in the current window, expires stale invites, closes stale bookings.
- `api/cron-storage-sweep.mjs` — daily. Deletes orphaned Supabase Storage objects (uploads whose owning row was deleted or migrated).

Both use `SUPABASE_SERVICE_ROLE_KEY` for admin queries.

## 18. CI / CD

**Hosting:** Vercel with Git integration. Push to `main` → production deploy. Preview URLs on every branch.

**Build:** `npm run build` = `build:api && tsc --noEmit && vite build`. Any tsc error fails.

**Predeploy hook:** `build:api` runs `scripts/build-api.mjs` → transpiles `api/_src/*.ts` to `api/*.mjs`. Vercel's function validation runs BEFORE `buildCommand`, so the `.mjs` outputs must be committed (documented pitfall — see memory `feedback_cadence_api_mjs.md`).

**No separate CI runner.** Tests currently not blocking; run locally.

**Rollback:** Vercel's built-in "Promote previous deployment".

## 19. Env / secrets management

- Local: `.env` at repo root. Untracked. Loaded by Vite (client vars via `VITE_*` prefix) and by `vercel dev` (all vars).
- Production: Vercel project env vars, one per environment (Preview, Production).
- **No secrets manager** (Doppler / Infisical / etc.).

## 20. Cost estimate (order-of-magnitude, for a 100-active-user MVP)

- **Vercel Hobby (personal projects) or Pro ($20/mo):** builds + hosting + serverless functions + edge middleware. Pro if crossing the 100GB bandwidth / 100k function invocations.
- **Supabase Pro ($25/mo):** 8 GB DB, 100 GB egress, unlimited API requests, daily backups.
- **Anthropic:** metered. Food estimates + AI drafts at Claude Haiku 4.5 pricing. For 100 users × ~10 estimates/day: ~$5/month.
- **VAPID web push:** free (self-signed).
- **Total:** ~$50/month for a 100-user MVP.
